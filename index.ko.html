How Track works

계획이 바뀌어도, 프로젝트는 앞으로 갑니다.

Goal / 01

계획은 바꿔도, 목표는 바꾸지 않습니다.

작업 순서와 실행 단위는 상황에 맞게 조정하되, 처음 정한 결과는 놓치지 않습니다.

Dependency / 02

막히지 않는 일부터 병렬로 돌립니다.

독립적인 작업은 동시에 진행하고, 함께 확인해야 할 일은 검증 결과가 모일 때까지 기다립니다.

Adapt / 03

필요한 만큼 나누고, 필요할 때 합칩니다.

Codex session은 작업 흐름에 맞춰 나누고 합치며, 필요하면 멈추거나 교체합니다.

Track in action

온보딩 베타를 흐름 끊김 없이 출시하세요.

Ask Codex $track으로 온보딩 베타 출시를 진행해 주세요. 끝난 작업은 재사용하고, 서로 막지 않는 일은 병렬로 진행하세요. 하나의 작업 맵을 계속 갱신하면서 출시 조건을 모두 확인할 때까지 이어가 주세요.

    and 성공 지표
        G->>M: 완료한 지표 작업 재사용
        M-->>I: 지표 검증 결과 전달
    and 지원 준비
        G->>S: 지원 준비 시작
        S-->>I: 준비 상태 전달
    end
    Note over P,S: 막히지 않는 작업은 병렬 진행
    I->>I: 출시 조건 모두 점검
    alt 점검 통과
        I-->>G: 베타 출시 승인
    else 문제 발견
        I-->>P: 문제 있는 작업만 다시 진행
        P-->>I: 검증 결과 다시 전달
    end

각 레인은 목표·제품·지표·지원처럼 역할별 작업 흐름을 보여줍니다. 세 작업의 검증 결과가 출시 점검에 모두 모여야 베타를 내보낼 수 있습니다. 문제가 생기면 전체가 아니라 해당 작업만 다시 확인합니다.

Scope control

할 일을 늘리기 전에, 지금 필요한지 먼저 봅니다.

새 작업이나 범위 변경은 현재 목표에 실제로 필요한지 먼저 확인합니다. 좋은 아이디어라도 이번 릴리스에 필요 없다면 기록만 남기고 미룹니다. Track은 할 일을 늘리기보다, 끝낼 수 있는 범위를 지킵니다.

  • Start지금 필요한 최소 범위로 시작합니다.
  • Trim더 작게 나눠 다시 판단합니다.
  • Hold기록은 남기고, 이번에는 하지 않습니다.
